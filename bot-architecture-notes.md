# Go Trading Bot — Architecture Notes

Reference notes for the bot project. Covers stack decisions, project layout, and key Go/HTTP concepts.

## HTTP: Client vs Server (Don't Mix These Up)

Both live in Go's `net/http` package, but they play opposite roles.

| Term | Role | When You Use It |
|---|---|---|
| **`http.Client`** | Makes outbound requests | Your bot calling Binance — most of your code |
| **`http.ServeMux`** ("mux") | Receives incoming requests, routes them to handlers | Only if you build a dashboard later |

```go
// CLIENT side — what your bot does to call Binance
resp, err := http.Get("https://api.binance.com/api/v3/ticker/price?symbol=BTCUSDT")

// SERVER side — only if you add a dashboard later
mux := http.NewServeMux()
mux.HandleFunc("GET /trades", handleTrades)
http.ListenAndServe(":8080", mux)
```

The bot itself runs as a **long-running process** on a cloud server (e.g. Hetzner VPS). It is *not* a browser app. Trading logic must never run in React — secrets would leak and the browser can't run 24/7.

## Stack Decisions

| Layer | Choice | Why |
|---|---|---|
| Language | Go 1.22+ | Fast, single binary, great concurrency for streams |
| HTTP | `net/http` standard library | Go 1.22 added pattern matching; no framework needed for personal scale |
| Binance SDK | `github.com/adshao/go-binance/v2` | Handles auth signing, REST + WebSocket |
| Database | SQLite via `modernc.org/sqlite` | Single file, pure Go, zero ops |
| SQL helper | `sqlc` | Generates type-safe Go from SQL — better than an ORM |
| Migrations | `golang-migrate` | Standard tool, simple |
| Task runner | `Task` (taskfile.dev) | Replaces Make; YAML-based |
| Config | `.env` file + `os.Getenv` (or `godotenv`) | Simple, secrets stay out of git |

Skip Gin, chi, and Postgres until they hurt. They probably won't at this scale.

## Monorepo Tooling (Like Nx) — Probably Skip For Now

Options ranked by overhead:

1. **Folders + Taskfile + Go workspaces (`go.work`)** — recommended. Native Go answer, zero extra tooling.
2. **Turborepo** — JS-native, can shell out to Go. Good if frontend grows.
3. **Nx with `@nx-go/nx-go` plugin** — supports Go via community plugin. Use only if you already love Nx.
4. **Bazel** — overkill for personal scale.

For a side project: start with one Go module + a `Taskfile.yml`. Add a `web/` folder when you actually want React. You can layer Nx or Turbo on top later — the directory structure below works with any of them.

## Directory Layout

```

├── go.mod                      # Go module root
├── go.sum
├── go.work                     # only if you split into multiple modules later
├── Taskfile.yml                # task runner (replaces make)
├── .env.example                # template; never commit .env
├── .gitignore
├── README.md
│
├── cmd/                        # entry points — one folder per binary
│   └── bot/
│       └── main.go             # `go run ./cmd/bot` starts the bot
│
├── internal/                   # private code — Go forbids external imports
│   ├── config/
│   │   └── config.go           # load API keys, settings from env
│   │
│   ├── exchange/               # everything Binance-related
│   │   ├── binance.go          # wraps go-binance SDK
│   │   ├── stream.go           # WebSocket price streams
│   │   └── types.go            # Order, Candle, Position structs
│   │
│   ├── strategy/               # your trading logic
│   │   ├── strategy.go         # Strategy interface
│   │   └── sma_crossover.go    # one concrete strategy to start
│   │
│   ├── risk/
│   │   └── manager.go          # position sizing, stop-loss, max drawdown
│   │
│   ├── ai/
│   │   └── client.go           # your AI provider (Claude, OpenAI, whatever)
│   │
│   ├── storage/
│   │   ├── sqlite.go           # DB connection
│   │   ├── queries.sql         # raw SQL for sqlc to generate from
│   │   ├── db.go               # generated by sqlc — don't edit
│   │   └── migrations/
│   │       ├── 001_init.up.sql
│   │       └── 001_init.down.sql
│   │
│   └── server/                 # OPTIONAL — only if you add a dashboard
│       ├── server.go
│       └── handlers.go
│
├── scripts/
│   └── seed.go                 # one-off scripts (backtest seed data, etc.)
│
└── web/                        # FUTURE React dashboard — leave empty for now
    ├── package.json
    ├── vite.config.ts
    └── src/
```

### Why this shape

- **`cmd/` and `internal/` are Go conventions, not arbitrary.** The toolchain enforces `internal/` — code inside it cannot be imported by other modules. Internals stay internal.
- **Group by domain, not by layer.** NestJS-style "controllers / services / repositories" doesn't fit Go culture. Go groups by domain (`exchange`, `strategy`, `risk`) and each folder owns its full vertical. Less ceremony.
- **`web/` lives next to `cmd/`, separately deployable.** Run `npm install` inside `web/`. The Go bot doesn't know or care about it. When you wire up the dashboard, `internal/server/` exposes a JSON API and React fetches from it. Decoupled.
- **No `pkg/` folder.** Only add it if you decide to publish reusable libraries. Personal projects don't need it.
